## 1. 컨테이너와 hypervisor

### VM 구조

VM은 하드웨어를 가상화

하이퍼바이저(Hypervisor)란? 가상 머신(Guest OS)을 생성·격리·관리하기 위한 소프트웨어 계층

하이퍼바이저가 하는 일:

CPU 가상화
메모리 분배
디스크 가상화
네트워크 가상화

Guest OS 격리
VM 생성/삭제/중지

즉 실제 하드웨어를 여러 Guest OS가 나눠 쓰게 해주는 관리자 역할

```
Hardware
 └─ Hypervisor
     └─ Guest OS
         └─ App
```

### Container 구조

컨테이너는 호스트 커널을 공유합니다.

```
Hardware
 └─ Host OS (Linux Kernel)
     └─ Container Runtime
         └─ Containers
```
컨테이너는 **Linux kernel의 namespace, cgroups, overlay filesystem** 을 공유한다.
os 를 따로 부팅하지않는다. 

- linux kernel의 namespace란?
    linux에서는 모든 프로세스가 PID, 네트워크, 파일시스템, hostname 등을 공유함.
    그런데 namespace로 격리하면 각 namespace에서는 

    - 같은 PID 사용 가능
    - 서로 다른 hostname 사용 가능
    - 서로 다른 network interface 사용 가능
    - 서로 다른 mount filesystem 사용 가능

    | namespace | 역할             |
    | --------- | -------------- |
    | PID       | 프로세스 격리        |
    | NET       | 네트워크 격리        |
    | MNT       | 파일시스템 mount 격리 |
    | UTS       | hostname 격리    |
    | IPC       | 프로세스 간 통신 격리   |
    | USER      | 사용자 ID 격리      |


    예시 ) Slurm Job 격리

    Slurm에서 사용자별 Job Container를 실행할 때:

    USER namespace
    NET namespace
    MNT namespace

    를 사용하면 사용자 Job끼리 독립 환경 제공 가능하다.

    예를 들어:

    User A Job
    User B Job

    이 서로 다른 namespace를 사용하면 네트워크 충돌 방지, 라이브러리 충돌 방지, 파일 접근 격리가 가능하다.


- cgroups(control groups)이란?
    CPU/메모리 얼마나 쓸지 제한하는 기능
    컨테이너 하나가 메모리를 100GB 정도 혼자 쓰는 경우를 막기위해, 
    cgroups 에 CPU, 메모리, I/O, 프로세스 갯수 등을 제한함.

    예시 ) Slurm HPC Cluster 자원 제한

    Slurm에서는 Job별로:

    CPU Core
    Memory
    GPU Count

    를 제한한다.

    예:

    #SBATCH --cpus-per-task=16
    #SBATCH --mem=64G
    #SBATCH --gres=gpu:4

    실제로 내부적으로는 Linux cgroups를 이용해:

    CPU quota
    memory limit
    device access

    를 제한한다.

    즉 Slurm의 Resource Scheduling 핵심 기술 중 하나가 cgroups이다.

- overlay filesystem이란?

    컨테이너 이미지는 보통 여러 Layer로 구성된다. 
    
    예를 들면 ubuntu base위에 python, flask, app을 빌드할 때 overlay filesystem은 이 layer를 겹쳐서 하나처럼 보이게 한다.

    아래와 같이

    Layer 1: Ubuntu Base
    Layer 2: Python 또는 Nginx 설치
    Layer 3: Flask 및 사용자 애플리케이션

    각 Layer는 실제로는 분리되어 저장되지만, 컨테이너 내부에서는 하나의 통합된 파일시스템처럼 보인다.

    OverlayFS의 중요한 특징은 다음과 같다.

    - Layer 재사용 가능
        Ubuntu Base Layer 같은 공통 Layer를 여러 컨테이너가 함께 공유할 수 있다.
        따라서 저장 공간을 효율적으로 사용할 수 있다.
    - 빠른 컨테이너 생성
        새 컨테이너를 만들 때 전체 파일을 복사하지 않는다.
        기존 Layer 위에 새로운 writable Layer만 추가하므로 컨테이너를 매우 빠르게 생성할 수 있다.
    - Copy-on-Write 방식 사용
        기존 Layer는 읽기 전용(Read-Only)으로 유지된다.
        컨테이너 내부에서 파일을 수정하면 변경된 부분만 새로운 Layer에 저장된다.

    즉 Overlay Filesystem은 컨테이너 이미지의 Layer를 효율적으로 관리하고, 저장 공간 절약과 빠른 배포를 가능하게 하는 핵심 기술이다.
